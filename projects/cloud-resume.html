Cloud Resume Challenge

A fully serverless, cloud-native resume site — built on AWS from static hosting to API-driven visitor counter, deployed with Infrastructure as Code and automated via CI/CD pipelines.

Status Live
Stack S3, CloudFront, Lambda, DynamoDB, Terraform, GitHub Actions
Role Solo

What it is

The Cloud Resume Challenge is a well-known rite of passage for aspiring cloud engineers. The challenge pushes you to build a real, production-style deployment — not a tutorial, not a sandbox. Every service has to actually talk to each other, deployed to your own AWS account.

My implementation goes end-to-end: a hand-coded HTML/CSS resume hosted on S3, served through CloudFront with HTTPS, backed by a Python Lambda function that reads and writes a visitor counter to DynamoDB, all wired up via API Gateway.

Architecture

Browser → CloudFront (CDN + HTTPS) ↓ S3 Bucket (static files) ↓ [Visitor Counter Button] ↓ API Gateway (REST) ↓ Lambda (Python) ↓ DynamoDB (counter)

What I built

  • Static resume in HTML/CSS, no frameworks
  • S3 bucket configured for static website hosting with proper bucket policies
  • CloudFront distribution with HTTPS via ACM certificate and custom domain
  • Python Lambda function for GET/POST visitor count via DynamoDB
  • API Gateway wired to Lambda with CORS configured correctly
  • Terraform modules for all infrastructure — no clicking through the console
  • GitHub Actions pipeline: push to main → tests → Terraform plan/apply → deploy

What I learned

The Cloud Resume Challenge is deceptively hard. Getting each AWS service to work in isolation is one thing. Getting them all to talk to each other — with correct IAM permissions, CORS headers, and Terraform state — is another level.

The biggest unlocks were understanding IAM least-privilege in practice, how CloudFront cache invalidation works, and how to structure Terraform so it's actually maintainable.
